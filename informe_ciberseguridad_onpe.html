Resumen Ejecutivo

Se realizó un análisis de seguridad defensivo sobre la API pública de resultados electorales de la Oficina Nacional de Procesos Electorales (ONPE) del Perú, desplegada durante las Elecciones Generales 2026. El sistema es accesible sin autenticación y sirve datos electorales en tiempo real a la ciudadanía.

Los hallazgos más críticos son la exposición del endpoint de padrón electoral (/padron/mesa/{dni}) sin ningún control de acceso ni rate limiting, y la documentación técnica completa de la API (Swagger UI + OpenAPI 3.1) accesible públicamente, que expone 65 endpoints internos, nombres de usuarios de sistema y URLs de entornos de desarrollo.

Se descubrieron además 30+ endpoints no documentados oficialmente, incluyendo endpoints para todas las elecciones del proceso (Diputados, Senadores, Parlamento Andino), endpoints de administración de caché y el endpoint de mapa de calor del frontend.

Ejemplo en Python:

import urllib.request, json

HEADERS = {
    'user-agent': 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/147.0.0.0 Safari/537.36',
    'sec-fetch-mode': 'cors',
    'sec-fetch-site': 'same-origin',
    'accept': '*/*',
}

def get(url):
    req = urllib.request.Request(url, headers=HEADERS)
    with urllib.request.urlopen(req, timeout=30) as r:
        return json.loads(r.read().decode('utf-8'))

Hallazgos de Seguridad

H1 Control de acceso basado en headers HTTP manipulables Media CVSS 5.3 · AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N

Descripción

El único mecanismo que diferencia un navegador legítimo de un bot es la presencia del header sec-fetch-mode: cors. Este header puede ser incluido libremente por cualquier script automatizado. No existe autenticación real de ningún tipo. La cookie _ga_* de Google Analytics tampoco es requerida para acceder a los datos.

Evidencia

# Acceso completo a la API sin cookie ni autenticación real:
curl "https://resultadoelectoral.onpe.gob.pe/presentacion-backend/eleccion-presidencial/participantes-ubicacion-geografica-nombre?idEleccion=10&tipoFiltro=eleccion" \
  -H "user-agent: Mozilla/5.0" \
  -H "sec-fetch-mode: cors" \
  -H "sec-fetch-site: same-origin"

# → HTTP 200  |  JSON con todos los votos nacionales
# Verificado: sec-fetch-site con valor "cross-site" también funciona (no validado)
# Verificado: referer con cualquier dominio funciona (no validado)

Mitigación

  • Implementar API key o token CSRF real validado en el backend Spring Boot
  • Rate limiting por IP en CloudFront WAF como control compensatorio
  • Validar el header Origin contra una whitelist en la lógica del backend (actualmente CORS está bien configurado a nivel de respuesta, pero no bloquea el acceso)
H2 Ausencia de rate limiting en todos los endpoints Media CVSS 5.8 · AV:N/AC:L/PR:N/UI:N/S:C/C:L/I:N/A:N

Descripción

No existe ningún límite de velocidad en la API. 10 requests consecutivos respondieron uniformemente entre 33–52ms sin throttling, HTTP 429 ni CAPTCHA.

Evidencia

req01: HTTP 200 — 42ms
req02: HTTP 200 — 40ms
req03: HTTP 200 — 39ms
...
req10: HTTP 200 — 41ms
# Sin degradación, sin bloqueo, sin aviso

Impacto combinado con H3

A 50ms por consulta, un único cliente puede realizar ~72,000 consultas/hora. Aplicado al endpoint /padron/mesa/{dni}, permitiría correlacionar masivamente el padrón de 27.3 millones de electores en pocas horas.

Mitigación

  • Rate limiting por IP en WAF de CloudFront: máx. 60 req/min para endpoints de datos generales
  • Rate limiting severo en endpoints sensibles: máx. 10 req/min en /padron/mesa/ y /actas/
  • Detección de patrones de enumeración secuencial de parámetros numéricos (DNIs, ubigeos)
H3 /padron/mesa/{dni} sin autenticación ni rate limiting Alta CVSS 7.5 · AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N

Descripción

El endpoint devuelve la mesa de votación asignada a cualquier ciudadano peruano dado su DNI, sin ningún control de acceso ni autenticación.

Evidencia

GET /presentacion-backend/padron/mesa/10001088
→ HTTP 200
→ {"success":true,"data":{"dni":"10001088","mesa":"059318"}}

GET /presentacion-backend/padron/mesa/07845838
→ HTTP 200
→ {"success":true,"data":{"dni":"07845838","mesa":"047934"}}

# DNIs de candidatos presidenciales (documentos públicos del JNE)
# usados para verificar el funcionamiento sin enumeración masiva

Impacto

La combinación DNI + mesa de votación permite:

  • Inferir el domicilio aproximado del elector (cada mesa corresponde a un local en un distrito específico)
  • Construir listas de votantes por zona geográfica a partir de bases de DNIs filtradas previas
  • Habilitar prácticas de clientelismo electoral dirigido o intimidación por zona
  • Correlacionar masivamente el padrón de 27.3 millones de electores sin ningún obstáculo técnico
Estado actual: El endpoint devuelve HTTP 204 (sin contenido) post-elección, indicando que está inactivo fuera del período electoral. Sin embargo, estuvo activo durante el día de la elección y el período de cómputo.

Mitigación (urgente para el próximo proceso electoral)

  • Requerir autenticación del propio elector: validación de DNI + fecha de nacimiento antes de devolver la mesa
  • Rate limiting severo: máx. 3 consultas/IP/hora, independientemente de la autenticación
  • Auditar los logs de acceso del endpoint para detectar si ocurrió enumeración masiva durante el proceso
  • Evaluar si el endpoint debe existir en la API pública o solo en el portal autenticado de onpe.gob.pe

Impacto

  • Cualquier atacante descubre y prueba endpoints administrativos sin reverse engineering del JS
  • Los nombres de usuarios internos son útiles para ingeniería social o credential stuffing en otros sistemas de ONPE
  • La URL de desarrollo permite inferir convenciones de nomenclatura de infraestructura interna

Mitigación

  • Deshabilitar en producción: springdoc.api-docs.enabled=false y springdoc.swagger-ui.enabled=false en application-prod.properties
  • Restringir Spring Actuator a IPs internas: management.server.address=127.0.0.1
  • Filtrar campos de auditoría de respuestas públicas con @JsonIgnore en entidades o usando DTOs sin esos campos
  • Agregar regla en CloudFront WAF para bloquear /swagger-ui/* y /v3/* desde IPs externas como medida de defensa en profundidad

H7 CSP con unsafe-inline / unsafe-eval y nonce no inyectado Baja CVSS 4.3 · AV:N/AC:L/PR:N/UI:R/S:U/C:L/I:N/A:N

7a — CSP con directivas inseguras

Content-Security-Policy: script-src 'self' 'unsafe-inline' 'unsafe-eval' https://code.createjs.com ...
                         style-src  'self' 'unsafe-inline' ...

7b — Nonce CSP no reemplazado (bug de infraestructura)

El HTML del frontend contiene el placeholder literal sin sustituir:

<meta name="csp-nonce" content="NGINX_CSP_NONCE">
<script nonce="NGINX_CSP_NONCE">...</script>

CloudFront sirve el HTML estático directamente desde S3 sin pasarlo por NGINX, por lo que el nonce nunca se inyecta. La directiva nonce de la CSP queda inoperativa, dejando unsafe-inline como único mecanismo activo para scripts inline.

Mitigación

  • Asegurar que el HTML pase por NGINX antes de ser servido por CloudFront, para que el nonce se reemplace correctamente en cada request
  • Reemplazar unsafe-inline con nonces CSP generados por request en NGINX
  • Evaluar si unsafe-eval es realmente necesario para las librerías incluidas (amCharts, CreateJS)

Recomendaciones Priorizadas

⚠ Urgente — Antes del próximo proceso electoral
  1. [H3] Autenticar el endpoint de padrón: implementar validación de DNI + fecha de nacimiento antes de devolver la mesa asignada. Mientras tanto, activar rate limiting severo (5 req/IP/hora). Auditar logs existentes para detectar si ocurrió enumeración masiva durante el proceso 2026.
  2. [H4] Deshabilitar Swagger en producción: agregar springdoc.api-docs.enabled=false y springdoc.swagger-ui.enabled=false en application-prod.properties. Agregar regla WAF en CloudFront para bloquear /swagger-ui/* y /v3/* como defensa en profundidad.
Corto plazo — 1 a 2 semanas
  1. [H4] Filtrar campos de auditoría: eliminar audUsuario* y audFecha* de todas las respuestas públicas usando DTOs o @JsonIgnore en Spring Boot.
  2. [H2] Rate limiting: implementar reglas en CloudFront WAF — máx. 60 req/min por IP en endpoints generales; máx. 10 req/min en endpoints de datos de padrón y actas individuales.
  3. [H5] Limitar paginación: añadir @Max(100) al parámetro tamanio en todos los endpoints paginados y devolver HTTP 400 si se supera.
Mediano plazo — Próximo ciclo de despliegue
  1. [H7] Corregir nonce CSP: revisar el pipeline de entrega del HTML para asegurar que NGINX procese y reemplace el nonce antes de que CloudFront lo sirva. Eliminar unsafe-inline usando nonces por request.
  2. [H6] Actualizar HSTS: cambiar a Strict-Transport-Security: max-age=31536000; includeSubDomains; preload y registrar en hstspreload.org.
  3. [H1] Control de acceso real: evaluar implementación de API key pública (sin autenticación de usuario) para identificar y limitar clientes automatizados sin afectar el acceso ciudadano.
  4. [H4] Restringir Actuator: configurar management.server.address=127.0.0.1 para que el health check solo sea accesible desde la red interna.
